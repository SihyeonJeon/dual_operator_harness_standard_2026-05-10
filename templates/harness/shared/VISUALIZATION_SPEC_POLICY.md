# Visualization Specification Policy

Visualization is a control surface, not decoration. Do not build a dashboard,
timeline, graph, local evidence page, manager view, or live status UI until the
pre-visualization specification gate passes.

## Authority

Canonical sources remain the file-backed harness artifacts:

- `feature_list.json`
- `progress.md`
- `session-handoff.md`
- `harness/shared/`
- `harness/tasks/`
- `harness/events/events.jsonl`

HTML, dashboards, graphs, timelines, and local evidence pages are compiled views over
those sources. They must not become competing memory or decision authority.

## Design Ownership

Claude owns visualization and diagram information architecture. A Claude Code
operator or Claude-backed design worker should draft or review task-local
`VISUALIZATION_SPEC.md` before visual production starts. Codex or another
deterministic worker may implement event plumbing, static HTML generation, tests,
and backend adapters after the approved Claude visualization spec exists.

Bitmap image generation is Codex-owned by default. When a task needs product
photos, mock photographs, hero images, raster illustrations, or other generated
bitmap assets, route that asset request through Codex image generation and record
the prompt, output path, review owner, and license/publication decision in the
task artifact. Claude Code owns diagrams, dashboards, timelines, graphs, report
information architecture, and visual explanation structure.

The design review must not force consensus between fixed operators. It is a
specialized design artifact that operators can accept, reject, or return for
revision with evidence.

## Required Pre-Visualization Spec

Before visualization production starts, create a task-local
`VISUALIZATION_SPEC.md` from `harness/templates/VISUALIZATION_SPEC.md` and
record:

- purpose and decision the visualization supports;
- intended audience and operator/human control surface;
- source artifacts and event streams;
- data contract and required fields;
- task, worker, operator, gate, and blocker states to show;
- views, filters, drilldowns, and empty/error states;
- redaction and privacy rules;
- update cadence: static, refresh-on-command, or live;
- acceptance criteria and evidence path;
- approval status.
- selected visualization backend:
  - `local_file` for dependency-free local status/evidence payloads;
  - project-selected external backend only after human approval.
- backend mode: static, refresh-on-command, live, external dashboard, or
  local evidence export.
- adapter owner and smoke evidence path when `events.jsonl` will flow to a
  backend.

## Gate

Visualization production is blocked until:

- planning confirms the visualization is needed for the active slice;
- design confirms the information architecture and state model;
- Claude visualization/design review exists for diagrams, dashboards, timelines,
  graphs, local evidence views, manager views, or live status UI;
- evaluation defines how the visualization will be checked;
- any redaction or external sharing risk is recorded;
- the current task artifact links the approved `VISUALIZATION_SPEC.md`.

If the spec is missing or materially ambiguous, return `SPEC_BLOCKED`.

## Report Rule

Static HTML reports generated by `scripts/harnessctl.py report` are allowed as a
local compiled view because they are derived from canonical files. They still
must state their source files and must not claim authority over the source
artifacts.

`scripts/harnessctl.py viz-export --backend local_file` is allowed as a local
adapter that converts `harness/events/events.jsonl` into sanitized files under
`harness/reports/viz/`. Any non-local viz backend is denied until the task-local
spec selects it and human-approved bounded policy, credential lifecycle records,
and smoke evidence exist.
